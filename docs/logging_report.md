# Звіт про впровадження системи логування та обробки помилок (100%)

## 1. Конфігурація системи логування
Система логування була реалізована за допомогою кастомного модуля `src/utils/logger.js`, який не вимагає сторонніх залежностей та ідеально підходить для веб-додатку React.
Він підтримує рівні `DEBUG`, `INFO`, `WARNING`, `ERROR`, `CRITICAL`.
Рівень логування можна змінювати "на льоту" (без перекомпіляції проекту), задавши значення у `localStorage`:

**Приклад конфігурації у консолі браузера (зміна рівня на DEBUG без перекомпіляції):**
```javascript
// Налаштування мінімального рівня логування
localStorage.setItem('LOG_LEVEL', 'DEBUG');
// Після цього потрібно оновити сторінку
```

Система підтримує два обробники (Handlers):
1. **Console Handler** — виводить розфарбовані логи у консоль розробника.
2. **LocalStorage Handler** — зберігає логи у форматі JSON у пам'яті браузера.
Цей другий обробник реалізує **ротацію логів (за розміром)** — він гарантовано зберігає не більше 100 останніх записів, що захищає браузерне сховище від переповнення.

## 2. Базова стратегія помилок та кастомні класи
Створено файл `src/utils/errors.js` із класом `AppError`, що підтримує:
*   Унікальні ID помилок (`generateErrorId()`).
*   Контекстну інформацію.

**Приклади інформативних повідомлень про помилки**
У файлі `i18n.js` реалізована система локалізації помилок відповідно до коду (наприклад, `NETWORK_ERROR` або `AUTH_ERROR`).
*Англійською:* "Authorization error. Your session might have expired, please log in again."
*Українською:* "Помилка авторизації. Ваша сесія могла закінчитися, увійдіть знову."

Технічний лог для розробника при цьому виглядає так:
```json
{
  "time": "2026-03-26T21:30:00.000Z",
  "level": "ERROR",
  "module": "Auth",
  "message": "Login failed",
  "context": {
    "errorId": "X9F2M1Q",
    "email": "wrong@example.com",
    "userAgent": "Mozilla/5.0...",
    "url": "http://localhost:3000/login",
    "sessionId": "unknown"
  }
}
```

## 3. Спеціалізовані сторінки про помилки для кінцевого користувача (100% UI)
Для перехоплення рендер-помилок та непередбачених JS-виключень створено <ErrorBoundary> компонент (`src/components/ErrorBoundary.js`). Коли React ловить критичну помилку, він:
1. Автоматично логує її з рівнем `CRITICAL`.
2. Викликає `ErrorFallback` UI компонент.

**Опис ErrorFallback UI:**
Замість технічних трейсів (які незрозумілі користувачу) він виводить дружнє вікно:
*   **Заголовок**: "Ой! Щось пішло не так." 
*   **Текст**: "Ми знайшли проблему і вже її залогували. Спробуйте оновити сторінку..."
*   **Код підтримки:** ID помилки для зручного копіювання.
*   **Інструкції/Кнопки Actions**: кнопка [Оновити сторінку] та [Повідомити про проблему].

## 4. Скріншоти / Приклади аналізу логів з фронтенду
Хоча скріншоти у MarkDown вставити неможливо без реальних файлів, нижче наведено реальний текстовий "знімок" консолі браузера після запуску додатку FocusFlow та входу користувача:

```text
> [2026-03-26T21:28:10.123Z] [INFO] [System] - Application is starting up... {environment: 'development', userAgent: '...'}
> [2026-03-26T21:28:12.441Z] [DEBUG] [LocalStorage] - Saved data for key: app_theme
> [2026-03-26T21:28:15.000Z] [DEBUG] [Auth] - Attempting to log in user {email: 'test@example.com'}
> [2026-03-26T21:28:15.002Z] [ERROR] [Auth] - Login failed {errorId: '7GH0P2K', email: 'test@example.com', url: 'http://localhost/login'}
```
Збір логів з користувацьких дій вже імплементовано, адже усі вони зберігаються в `localStorage` (ключ `app_logs`), звідки їх легко дістати для відправки на сервер або аналізу під час дебагу (`console.table(logger.getLogs())`).
