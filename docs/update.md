# Процедура оновлення (Update Guide) проєкту "My Planner"

Цей документ описує покрокові рекомендації для Release Engineer / DevOps щодо оновлення існуючого production-розгортання проєкту.

## 1. Підготовка до оновлення

Успішне оновлення вимагає ретельної підготовки:

### 1.1 Створення резервних копій (Бек-ап)
Перед початком оновлення обов'язково виконайте процедуру резервного копіювання. Усі необхідні інструкції знаходяться у файлі [backup.md](./backup.md).

### 1.2 Перевірка сумісності
- Перевірте `CHANGELOG.md` або Release Notes нової версії.
- Переконайтесь, що нова версія сумісна з поточною версією Node.js (якщо збірка на сервері) та конфігурацією Nginx.
- Перевірте сумісність змінних оточення (чи не додано нові обов'язкові змінні в `.env`).

### 1.3 Планування часу простою (Downtime)
Для SPA на основі Nginx час простою (Downtime) зазвичай становить секунди (фактично, час копіювання файлів і перезапуску вебсервера). 
Проте, **якщо планується оновлення API або міграція БД (у майбутньому), плануйте вікно технічного обслуговування (Maintenance Window) на години найменшої активності користувачів (наприклад, з 2:00 до 4:00 ночі).** Налаштуйте сторінку «Maintenance Area» у Nginx, якщо очікується понад 1 хв простою.

## 2. Процес оновлення

Після підготовки можна розпочати процес оновлення.

### 2.1 Зупинка потрібних служб
Для зменшення шансу часткового завантаження старих ассетів клієнтами можна тимчасово зупинити або перевести проєкт у maintenance-режим:
```bash
# Перенаправлення трафіку на maintenance-сторінку (якщо налаштовано в Nginx)
sudo systemctl stop nginx
```

### 2.2 Розгортання нового коду
1. Отримайте нову збірку:
   ```bash
   cd /tmp/my-planner-source
   git fetch origin
   git checkout tags/vX.X.X # або потрібну гілку
   npm ci
   npm run build
   ```
2. Замініть існуючу збірку:
   ```bash
   sudo rm -rf /var/www/my-planner/build
   sudo cp -r build /var/www/my-planner/
   sudo chown -R www-data:www-data /var/www/my-planner/build
   ```

### 2.3 Міграція даних та оновлення конфігурацій
- Якщо для нової версії потрібні нові змінні (API keys, URLs), оновіть `.env` файл перед збіркою на сервері або у CI-середовищі.
- *(Далi, при наявностi Backend)*: Запустіть міграції БД, напр.: `npm run db:migrate`.

- Перезапустіть веб-сервер, щоб застосувати нові файли:
  ```bash
  sudo systemctl start nginx # або reload/restart
  ```

## 3. Перевірка після оновлення

*(Примітка: Детальне тестування, моніторинг та траблшутинг винесені в окремі лабораторні роботи, тому тут описані коротко).*

- **Тести правильності роботи:** Переконайтесь, що сервіс віддає нову версію, не віддає кешований `index.html` (відкрийте Incognito Mode або очистіть кеш), а маршрутизація працює.
- **Моніторинг продуктивності:** Перевірте логи (`/var/log/nginx/access.log` та `error.log`) на предмет збільшеної кількості 404 чи 500 помилок.
- **Можливі проблеми та їх вирішення:** Якщо нові скрипти не завантажуються, перевірте права доступу (`chmod` та `chown`). Якщо виникли CORS-помилки – перевірте конфігурацію API-ендпойнтів у новому релізі.

---

## 4. Процедура відкату (Rollback)

У разі невдалого оновлення (Critical Bugs на Production, падіння відвідуваності через помилки 500), необхідно розпочати **Rollback**.

1. **Зупиніть веб-сервер для уникнення подальших збоїв у клієнтів:**
   ```bash
   sudo systemctl stop nginx
   ```
2. **Відновіть попередню робочу версію з бек-апу (див. `backup.md`):**
   ```bash
   # Видаліть "зламану" поточну збірку
   sudo rm -rf /var/www/my-planner/build
   
   # Перейменуйте (відновіть) бек-ап папку в робочу
   # (де YYYYMMDDHHMMSS - дата створеного бек-апу)
   sudo mv /var/www/my-planner/build_backup_YYYYMMDDHHMMSS /var/www/my-planner/build
   ```
3. **Відновіть попередні конфігурації/БД:** 
   Якщо `.env` файл був змінений, поверніть попередній з бек-апу. Якщо проводилась міграція бази даних, виконайте відкат (down-migration): `npm run db:rollback`.

4. **Перезапустіть сервіси та перевірте логі:**
   ```bash
   sudo systemctl start nginx
   ```
5. **Верифікація:** Перевірте у браузері (через Incognito), що завантажилася стара-стабільна версія. Якщо це так, відкат здійснено успішно. Повідомте команду розробників про невдалий реліз.
